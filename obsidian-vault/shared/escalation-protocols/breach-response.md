# Regulatory Ratio Breach Response Protocol

## Trigger Condition

This protocol activates when any of the following regulatory minimums is breached or when an agent's analysis indicates a breach is imminent within 5 business days:

| Ratio | Regulatory Minimum | Brooker Internal Trigger |
|-------|-------------------|-------------------------|
| LCR | 100% | < 100% (breach) or < 105% (imminent) |
| NSFR | 100% | < 100% (breach) or < 105% (imminent) |
| CAR | 8.0% (regulatory); 12.5% (internal) | < 12.5% (internal breach) |
| CET1 | 4.5% (floor); 7.0% (with buffer) | < 7.0% (distribution restrictions triggered) |
| Duration Gap | 3.0 years (internal) | > 3.0 years |
| Covenant ratio | Facility-specific (typically 4.0x ND/EBITDA) | Any covenant breach |

## Severity Level

- [x] Critical — emergency response, executive notification

All regulatory ratio breaches are classified Critical without exception. Agents may not downgrade severity based on context or management reassurance.

## Escalation Path

1. **Immediate (T+0):** Agent raises Critical escalation in Slack #cac-committee with full detail (ratio, current value, threshold, source evidence, timestamp).
2. **T+0 (simultaneous):** `email-notifier` sends HOD email with approval-ui link if a remediation proposal is ready, plus breach notification.
3. **T+1 hour:** If no HOD acknowledgement, `email-notifier` re-escalates to Deputy CEO / CFO.
4. **T+4 hours:** If no executive acknowledgement, escalate to Board Risk Committee Chair via email.
5. **T+24 hours (regulatory):** HOD or CFO must notify the Local Banking Authority of the breach per regulatory notification requirements. This is a legal obligation — the agent flags this deadline explicitly.

## Required Actions

**Immediate (agent responsibilities):**
- [ ] Calculate the exact breach magnitude: "LCR of 94.20% is 5.80pp below the 100% regulatory minimum."
- [ ] Identify the source evidence supporting the breach detection (document, page, Slack, date).
- [ ] Raise Critical escalation in Slack #cac-committee with the standard escalation format.
- [ ] Trigger email notification to HOD via email-notifier service.
- [ ] Log the breach event in the audit trail with timestamp, ratio, value, and source.
- [ ] Do NOT propose a staging change for the breaching ratio without explicit HOD instruction — management must direct the response.

**Within 1 hour (HOD responsibilities):**
- [ ] Acknowledge the escalation in Slack or approval-ui.
- [ ] Convene emergency CAC committee call or confirm async handling.
- [ ] Direct the relevant specialist agent to analyse remediation options.

**Within 4 hours (management responsibilities):**
- [ ] Determine the root cause of the breach (data error vs. genuine deterioration).
- [ ] If data error: obtain corrected source data; agent re-runs analysis; document correction.
- [ ] If genuine breach: develop remediation plan (asset sales, new funding, capital injection).
- [ ] Assign remediation actions with owners and deadlines.

**Within 24 hours (regulatory obligation):**
- [ ] HOD / CFO submits regulatory breach notification to Local Banking Authority as required by licence conditions.
- [ ] Document the notification submission in `/data/archive/` with timestamp and reference number.

**Ongoing (until resolved):**
- [ ] Daily monitoring of the breaching ratio — agent provides updated analysis each business day.
- [ ] Staging proposals for remediation steps follow the normal staging pipeline (require HOD approval).
- [ ] Agent tracks cure progress: "LCR has improved from 94.20% to 97.50% — 2.50pp below minimum; target resolution by [date]."

## Resolution Criteria

This protocol is considered resolved when ALL of the following are true:
1. The breaching ratio is confirmed to be at or above its regulatory minimum for two consecutive business days.
2. The root cause has been documented and signed off by the HOD.
3. If a genuine breach occurred: the regulatory notification has been submitted and acknowledged by the authority.
4. A post-incident review has been completed and any process improvements have been logged in the decisions vault.
5. The breach record has been archived to `/data/archive/` with full audit trail.

## Agent Notes

- When detecting a potential breach, always verify the source before raising Critical: confirm the document is current and the value is not a draft or forecast figure.
- If breach is detected from a Slack message only (not a formal document), raise a High escalation requesting confirmation — do not raise Critical on unverified informal communication alone.
- After raising the escalation, do not repeat it on every subsequent query — note that an open escalation exists and reference its ID.
- Agents must never advise management to delay regulatory notification — that is a legal decision for the HOD/CFO, but agents must clearly state the 24-hour deadline.
- If the breach turns out to be a data error, document the correction in the decisions vault with the source of the error and the corrected value.
