## Key design decisions

- **Hard rule preserved**: no service ever writes directly to the vault or to corporate data. All B3/B4/B5 outputs land in `/data/staging/pending/` for HOD approval via approval-ui. The existing data-zone discipline from `CLAUDE.md` is unchanged.
- **Two manifest schemas**: existing `ManifestProposal` (Excel cell change) is unchanged; new `VaultStagingManifest` (vault file write with `target_vault_path`, `operation`, `draft_content`, optional `synthesis_evidence` + `source_run_id` for batching). Both documented in `CLAUDE.md`.
- **LLM injection everywhere**: every LLM-calling function takes an `llm: LLMInvoker` parameter. Tests inject deterministic stubs; production uses `langchain_openai.ChatOpenAI`. Zero LLM dependency in the unit test suite.
- **Per-extractor failure isolation**: B3 fan-out uses `asyncio.gather` with try/except per extractor; one model hiccup never kills the whole batch.

## Open questions resolved in this PR

- **B4 per-dept thresholds** — committed defaults in `config/synthesis_thresholds.json` (regulations=2, finance=2, ic=3, research=4, macro=4, default=3). Editable per-dept.
- **B3 cadence** — `meeting_fanout_enabled` config flag (default on); per-dept opt-out is a one-line config change.
- **`proposal_source` flag** — added to `VaultStagingManifest`: `agent | human | vault_automation`. approval-ui can filter/group.
- **B5 `_memory/` semantics** — reflection-engine's existing `promoter.py` writes per-agent `memory.md`; B5 extends that pattern instead of introducing new schema.
